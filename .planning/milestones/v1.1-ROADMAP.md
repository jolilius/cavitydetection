# Milestone v1.1: Experiment Runs + Per-Step Analysis

**Status:** ✅ SHIPPED 2026-05-15
**Phases:** 3–4
**Total Plans:** 6

## Overview

v1.1 made the LLM's full reasoning trajectory first-class data. Instead of recording only the final best solution, every checkpoint in the evolution is now a queryable row — with the exact C source at that step and a per-checkpoint explanation of what changed since the previous checkpoint. Experiments are grouped as named runs so researchers can compare prompts that ran under identical conditions.

## Phases

### Phase 3: Experiment Run Structure

**Goal**: Researchers can group all prompts from one `make evolve-all` invocation as a single named run, and all legacy data is accessible in the new layout without data loss
**Depends on**: Phase 2
**Requirements**: RUNORG-01, RUNORG-02, RUNORG-03, MIGRATE-01, DISPLAY-01
**Plans**: 3 plans in 2 waves

Plans:

**Wave 1**
- [x] 03-01-PLAN.md — Test scaffolding + run_experiment.py run ID, --run/--output-root args, and metadata.json write
- [x] 03-02-PLAN.md — Migration script (migrate_legacy.py) for all legacy flat directories

**Wave 2**
- [x] 03-03-PLAN.md — Display scripts adaptation (show_results.py, show_consolidated.py) + Makefile RUN= pass-through and evolve-all run grouping

**Key decisions:**
- Run ID format: `YYYY-MM-DD_HHMM_<sanitized-model>` (colon/dot → dash); overridable with `--run`
- `metadata.json` uses read-then-merge pattern (idempotent prompt append)
- Path traversal guard: `os.path.abspath(run_dir).startswith(os.path.abspath(output_root))`
- Glob-based discovery in display scripts; `extract_run_id()` parses `runs` segment from path

**Success Criteria verified:**

### Phase 4: Per-Step Data Pipeline

**Goal**: Researchers can inspect the exact C source and an explanation of what changed at every checkpoint, loaded via the same `load_results()` / `load_all_results()` interface
**Depends on**: Phase 3
**Requirements**: CKPT-01, CKPT-02, CKPT-03, EXPLAIN-01, EXPLAIN-02
**Plans**: 3 plans in 2 waves

Plans:

**Wave 1**
- [x] 04-01-PLAN.md — Wave 0 checkpoint tests + checkpoint-based `_extract_iterations()` in `consolidate_results.py` + extended column list in `results_loader.py` (CKPT-01, CKPT-02, CKPT-03)

**Wave 2**
- [x] 04-02-PLAN.md — Per-checkpoint explanation loop in `run_experiment.py` (EXPLAIN-01, EXPLAIN-02)
- [x] 04-03-PLAN.md — Wave 0 regenerate test + `--regenerate` flag + `regenerate_results()` helper in `migrate_legacy.py` (D-09, D-10)

**Key decisions:**
- Checkpoint directory scan with numeric sort (`int(d.replace("checkpoint_", ""))`) — prevents lexicographic ordering bugs
- `iteration` alias preserved for backward compatibility with Phase 1/2 `load_results()` callers
- Sliding-window baseline: checkpoint N compares to checkpoint N-1 (first checkpoint compares to `initial_program.c`)
- Dict keys for explanation lookup are integer folder Ns, not `info["iteration"]` (Pitfall 2)
- `--regenerate` flag is opt-in; existing migration behavior unchanged (D-09)
- `shutil.copy2` backup (`results.json.v1`) always created before overwrite (D-10)
- `time_score=null` intentional (D-01); `combined_score=mem_score` until per-checkpoint timing available (D-02)

**Success Criteria verified:**
1. `load_results()` returns one row per checkpoint with all Phase 4 fields — ✓ (12 rows from real baseline)
2. `code` column contains full compilable C source — ✓
3. Each explanation describes what changed since previous checkpoint — ✓ (live UAT 2026-05-15)
4. `load_all_results()` aggregates across runs/programs/prompts — ✓

**Status**: COMPLETE (2026-05-15)
